2.Backtrace

实验目标很简单：在 xv6 内核中实现一个 backtrace() 函数。当这个函数被调用时，它需要能够打印出当前的函数调用栈，也就是一系列的返回地址。

这个功能在调试中极为有用。当内核 panic（崩溃）时，如果能自动打印出 backtrace，我们就能清晰地看到崩溃前内核正在执行哪个函数，以及这个函数又是被谁一层层调用过来的，从而快速定位问题根源。

核心原理：理解栈帧 (Stack Frame)

要实现 backtrace，首先必须理解“栈帧”这个概念。每一次函数调用，都会在栈上创建一个属于自己的内存区域，这个区域就是该函数的“栈帧”。

在 RISC-V 架构下，一个关键的寄存器是 s0，它被用作栈帧指针 (Frame Pointer, fp)。fp 指向当前栈帧的某个固定位置。而实现回溯的关键，就藏在每个栈帧里：

  1. 返回地址 (Return Address, ra): 当函数执行完毕后，需要返回到哪里继续执行。这个地址被保存在相对于当前 fp -8 字节的位置。
  2. 上一个栈帧的指针 (Saved Frame Pointer): 每个栈帧里都会保存调用它的那个函数的 fp 地址。这个地址被保存在相对于当前 fp -16 字节的位置。

正是这个“上一个栈帧的指针”，将所有活跃的函数调用栈帧像链表一样串联了起来。我们的 backtrace() 要做的，就是从当前的 fp（s0 寄存器）出发，沿着这个链表，一层层地向上回溯。

(注：这张当时在 GDB 中观察内存的截图，非常直观地展示了栈帧链表的结构。彩色的 fp 列中的地址，指向了调用它的那个函数的栈帧地址，完美地形成了一个链条。)

调试路上的“坑”与心得

这个实验虽然代码不长，但我还是遇到了两个经典的“坑”，记录下来以备后忘。

坑一：“unrecognized opcode” 编译错误

我最初把 r_fp() 函数直接添加到了 kernel/riscv.h 中，结果编译时报了一连串的“无法识别的操作码”错误。

  • 原因: riscv.h 这个头文件不仅被 C 文件 (.c) 包含，还被一些汇编文件 (.S) 包含。汇编器完全不认识 C 语言的 static inline uint64 等语法，把它当成了汇编指令，自然就报错了。
  • 解决方案: 使用条件编译指令 #ifndef __ASSEMBLER__ ... #endif 将 C 函数包裹起来，这样它对汇编器就“不可见”了，问题解决。这是操作系统开发中一个非常常见且重要的技巧。

坑二：内核崩溃 (Kernel Panic)

我的第一个 backtrace 版本一运行就导致内核崩溃。

  • 原因: 在 while 循环中，我的边界检查条件不严谨，导致 fp 在回溯到最顶层栈帧后，又根据一个无效的地址继续“回溯”，最终试图读取一块非法内存，引发了致命的页错误。
  • 解决方案: 严格使用 PGROUNDDOWN(fp) 和 PGROUNDUP(fp) 来限定 fp 的有效范围，确保回溯链条不会“越界”。

3.Alarm

要求：本次实验的核心目标是为 xv6 添加一个新的系统调用 sigalarm(interval, handler)。这个函数允许一个用户进程设置一个“闹钟”：

  • interval：一个时间间隔，以“时钟滴答 (tick)”为单位。
  • handler：一个函数指针，指向一个用户态的函数。

当该进程累计在 CPU 上运行了 interval 个 tick 之后，内核会中断该进程的正常执行流程，转而去执行 handler 函数。当 handler 执行完毕后，程序需要能无缝地回到之前被中断的地方，继续执行。
